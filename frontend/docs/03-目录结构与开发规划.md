# 03 — 前端目录结构与开发阶段规划

> **目的**：给出可直接落地的前端目录结构，并将开发任务按依赖关系拆分为 7 个阶段。每个阶段有明确的可交付产出，让课程设计有一个逐步推进的节奏。

## 二、开发阶段规划（7 个阶段）

### 阶段划分总览

```
Phase 1 ─── 项目地基
  │
  └─► Phase 2 ─── 公开浏览（游客可完整使用）
        │
        └─► Phase 3 ─── 店铺详情（信息展示）
              │
              ├─► Phase 4 ─── 用户认证（身份体系）
              │     │
              │     ├─► Phase 5 ─── 互动操作（评论/评分/点赞/收藏）
              │     │
              │     └─► Phase 6 ─── 个人中心 + 店铺上传
              │
              └─► Phase 7 ─── 管理员后台 + 社区反馈
```

---

### Phase 1：项目地基（预计 1-2 天）

**目标**：搭建可运行的项目骨架，安装全部依赖，统一基础设施。

**具体任务**：
- 安装 Tailwind CSS + Headless UI，配置 `tailwind.config.js`
- 安装 Axios，创建 `api/client.ts`（baseURL + 请求/响应拦截器骨架）
- 安装 React Router，创建路由表骨架（先只有 Home 占位页）
- 安装 TanStack Query，在 `main.tsx` 挂载 `QueryClientProvider`
- 安装 Zustand，创建 `authStore.ts` 骨架
- 创建 `RootLayout`（Header + Footer + `<Outlet/>`）
- 创建基础 UI 组件：`Button`、`Input`、`Modal`

**交付物**：
- 浏览器打开 `localhost:5173` 能看到 Header + Footer + 占位首页
- Axios 实例已配置，可发送请求（后端未就绪时用 MSW 或 mock 数据代替）

**后端依赖**：无。此阶段可完全独立进行。

---

### Phase 2：公开浏览 —— 店铺列表、搜索、筛选（预计 2-3 天）

**目标**：实现游客浏览闭环 —— 从首页列表到搜索筛选，无需登录即可完整使用。

**涉及页面/组件**：
- `HomePage` — 店铺卡片列表 + 系统公告横幅
- `DiscoverPage` — 支持排序切换的全量列表
- `SearchPage` — 搜索结果
- `ShopCard`、`ShopList`、`SortBar`、`FilterBar`、`SearchBar`
- `Pagination`、`EmptyState`、`Skeleton`

**关键实现点**：
- 使用 `useQuery` 请求店铺列表，`queryKey` 包含排序/筛选/分页参数
- 搜索框使用 `useDebounce` 防抖（300ms 后发起请求）
- 品类/区域字典从后端获取后缓存（`staleTime: Infinity`）

**后端必须就绪的接口**：

| 方法 | 路径 | 说明 |
|:---|:---|:---|
| GET | `/api/shops` | 店铺列表（支持 `sort`, `category`, `area`, `page`, `page_size`） |
| GET | `/api/shops/search?q=keyword` | 店铺名称模糊搜索 |
| GET | `/api/dictionaries/categories` | 品类字典 |
| GET | `/api/dictionaries/areas` | 区域字典 |
| GET | `/api/announcements/active` | 当前有效公告 |

**交付物**：游客可浏览店铺列表、切换排序、按品类/区域筛选、搜索店铺，体验完整流畅。

---

### Phase 3：店铺详情页（预计 2-3 天）

**目标**：实现全站信息密度最高的页面 —— 店铺详情页，展示完整信息、菜单、图片、评论预览和问答预览。

**涉及页面/组件**：
- `ShopDetailPage`
- `RatingStars`（展示态）、`RatingDist`（评分分布条形图）
- `ImageGallery`（封面大图 + 环境图组）
- `CommentList`（只读）、`CommentCard`
- `QuestionList`（只读）、`QuestionCard`

**关键实现点**：
- 详情页每次进入时自动触发浏览记录（调用后端记录接口，静默不阻塞渲染）
- 菜单以表格形式展示（菜名 | 价格 | 描述 | 图片）
- 评分分布用纯 CSS 横向条形图（或轻量 SVG），不引入图表库

**后端必须就绪的接口**：

| 方法 | 路径 | 说明 |
|:---|:---|:---|
| GET | `/api/shops/:id` | 店铺完整详情（含菜单、图片、评分分布） |
| GET | `/api/shops/:id/comments` | 评论列表（分页） |
| GET | `/api/shops/:id/questions` | 问答列表（分页） |
| POST | `/api/history` | 记录浏览历史（登录用户） |

**交付物**：完整的店铺详情页，所有信息正确渲染，无登录门槛。

---

### Phase 4：用户认证 —— 登录/注册/权限拦截（预计 2 天）

**目标**：建立完整的用户身份体系，实现"游客浏览 + 登录拦截"设计模式。

**涉及页面/组件**：
- `LoginPage`、`RegisterPage`
- `AuthGuard`（路由级拦截 — 保护需登录的页面）
- `Modal`（操作级拦截 — 游客点击收藏/评论时弹出"登录后即可XXX"引导弹窗）
- `authStore`（Zustand 完善 — 登录/登出/Token 持久化）
- `api/client.ts`（完善拦截器 — 请求注入 Token + 响应处理 401）

**关键实现点**：
- **两种拦截模式**：
  - 路由级：`/favorites`、`/upload-shop`、`/notifications` 等页面使用 `AuthGuard` 包裹，未登录直接重定向至登录页
  - 操作级：游客在店铺详情页点击"收藏"或"写评论"，不跳页，弹出模态框引导注册/登录，登录成功后回到原页面
- 登录成功后存储 Token 至 localStorage，Zustand 同步更新状态
- 401 响应拦截：Token 过期自动清除登录态，Toast 提示"登录已过期，请重新登录"

**后端必须就绪的接口**：

| 方法 | 路径 | 说明 |
|:---|:---|:---|
| POST | `/api/auth/register` | 注册（邮箱或手机号 + 密码 + 昵称） |
| POST | `/api/auth/login` | 登录（邮箱或手机号 + 密码），返回 JWT Token |

**交付物**：完整的注册-登录-登出流程，Token 自动管理，游客与登录用户的权限边界清晰。

---

### Phase 5：互动操作 —— 评论/评分/点赞/收藏（预计 3-4 天）

**目标**：实现所有用户互动功能，每个写操作均配备乐观更新。

**涉及页面/组件**：
- `CommentForm`、`CommentList`（升级为可交互）
- `RatingStars`（交互态 — 点击评分）
- `QuestionForm`、`AnswerForm`、`QuestionList`（升级为可交互）
- `LikeButton`（乐观更新）
- `FavoriteButton`（乐观更新）

**关键实现点**：
- **乐观更新模式**（以收藏为例）：

  ```
  useMutation 的 onMutate：
    1. 取消相关查询的进行中请求
    2. 快照当前缓存
    3. 乐观修改缓存（切换收藏状态 ±1）
  onError：
    1. 回滚缓存至快照
    2. Toast 提示"操作失败，请重试"
  onSettled：
    1. 重新获取最新数据，确保与服务器一致
  ```

- **评分组件**：点击星级立即高亮显示，异步提交评分，失败回滚到原状。每用户每店只能评一次（后端返回 409 时前端 Toast "您已对该店铺评过分"）
- **评论配图上传**：表单中嵌入图片选择 + 压缩 + 进度条

**后端必须就绪的接口**：

| 方法 | 路径 | 说明 |
|:---|:---|:---|
| POST | `/api/shops/:id/rate` | 提交评分（1-5整数） |
| POST | `/api/shops/:id/comments` | 发表图文评论 |
| DELETE | `/api/comments/:id` | 删除自己的评论 |
| POST | `/api/comments/:id/like` | 点赞评论 |
| DELETE | `/api/comments/:id/like` | 取消点赞 |
| POST | `/api/shops/:id/questions` | 发起提问 |
| POST | `/api/questions/:id/answers` | 回答问题 |
| POST | `/api/shops/:id/favorite` | 收藏店铺 |
| DELETE | `/api/shops/:id/favorite` | 取消收藏 |

**交付物**：所有互动操作流畅可用，乐观更新体验合格（点击无延迟感），失败有提示。

---

### Phase 6：个人中心 + 店铺上传（预计 3-4 天）

**目标**：完成用户个人空间的所有页面，以及店铺创建与图片/菜单补充上传。

**涉及页面/组件**：
- `UserProfilePage`（查看 + 编辑模式切换）
- `UserPublicPage`（他人视角的用户动态聚合页）
- `FavoritesPage`（收藏夹列表，支持取消收藏）
- `HistoryPage`（浏览历史，支持清空/单条删除）
- `NotificationsPage`（通知列表 + 已读标记 + 点击跳转）
- `UploadShopPage`（极简三字段表单）
- `Avatar` 上传裁剪组件
- `ActivityFeed`

**关键实现点**：
- 收藏夹、浏览历史均为**列表 → 详情**的标准模式，可复用 `ShopCard`
- 通知列表按时间倒序，未读通知左侧有圆点标识，支持"全部已读"
- 上传店铺表单仅三字段（名称 + 区域下拉 + 品类下拉），名称输入框 onBlur 触发查重请求

**后端必须就绪的接口**：

| 方法 | 路径 | 说明 |
|:---|:---|:---|
| GET | `/api/users/me` | 获取自己的完整资料 |
| PUT | `/api/users/me` | 修改昵称/头像/简介/性别 |
| GET | `/api/users/:id/profile` | 查看他人公开资料 |
| GET | `/api/users/:id/activity` | 查看他人公开动态 |
| GET | `/api/favorites` | 收藏夹列表（按收藏时间倒序） |
| GET | `/api/history` | 浏览历史列表 |
| DELETE | `/api/history/:id` | 删除单条历史 |
| DELETE | `/api/history` | 清空历史 |
| GET | `/api/notifications` | 通知列表 |
| PUT | `/api/notifications/read-all` | 全部标记已读 |
| POST | `/api/shops` | 创建店铺（名称+区域+品类） |
| POST | `/api/shops/:id/check-name` | 名称查重（上传前校验） |
| POST | `/api/shops/:id/images` | 上传店铺图片/菜单 |

**交付物**：完整个人中心（资料+收藏+历史+通知），以及极简店铺上传能力。

### Phase 7：社区反馈 + 管理员后台（预计 4-5 天）

**目标**：实现社区自我净化机制的用户端 + 管理员后置治理中枢。**此阶段页面最多，建议最后集中开发**。

**涉及页面/组件**：
- `ReportModal`（用户侧 — 举报弹窗）
- 店铺详情页勘误反馈入口 + 重复店铺反馈入口
- `admin/DashboardPage`（数据概览卡片 + 7日趋势）
- `admin/ReportsPage`（举报队列 + 详情 + 通过/驳回操作）
- `admin/ShopManagePage`（店铺列表 + 封禁/解封）
- `admin/UserManagePage`（用户列表 + 封禁/解封）
- `admin/AnnouncementPage`（发布/编辑公告）
- `admin/ShopMergePage`（选择主店铺 + 合并确认）
- `AdminLayout`（后台侧边导航布局）
- `AdminGuard`（路由级管理员权限校验）

**关键实现点**：
- 管理员后台使用独立布局 `AdminLayout`，与前台 `RootLayout` 视觉区分
- 举报/勘误/重复反馈在同一个列表页中展示，通过类型筛选标签切换
- 封禁操作前弹出确认框

**后端必须就绪的接口**：

| 方法 | 路径 | 说明 |
|:---|:---|:---|
| POST | `/api/reports` | 提交举报 |
| POST | `/api/feedbacks/correction` | 提交店铺信息勘误 |
| POST | `/api/feedbacks/duplicate` | 提交重复店铺反馈 |
| GET | `/api/admin/dashboard` | 数据概览（总数+趋势） |
| GET | `/api/admin/reports` | 举报/反馈列表 |
| PUT | `/api/admin/reports/:id/handle` | 处理举报（通过/驳回） |
| PUT | `/api/admin/shops/:id/ban` | 封禁/解封店铺 |
| PUT | `/api/admin/users/:id/ban` | 封禁/解封用户 |
| POST | `/api/admin/shops/merge` | 合并店铺 |
| POST | `/api/admin/announcements` | 发布公告 |
| PUT | `/api/admin/announcements/:id` | 编辑公告 |

**交付物**：完整的管理员后台，以及用户侧的举报/反馈入口。

## 三、阶段依赖与后端对接时序

```
后端同学需要提前交付的接口批次：

  批次 A（Phase 2 前）  →  店铺列表/搜索/字典/公告     [约 5 个接口]
  批次 B（Phase 3 前）  →  店铺详情/评论浏览/问答浏览   [约 3 个接口]
  批次 C（Phase 4 前）  →  注册/登录                   [约 2 个接口]
  批次 D（Phase 5 前）  →  评分/评论/问答/点赞/收藏     [约 9 个接口]
  批次 E（Phase 6 前）  →  用户资料/收藏夹/历史/通知/店铺创建  [约 12 个接口]
  批次 F（Phase 7 前）  →  举报/反馈/管理员全部后台接口    [约 11 个接口]
```

**联调建议**：
- Phase 1 期间与后端同学确认接口文档（FastAPI 自动生成的 Swagger UI）
- 每个 Phase 开始前，先用 Swagger 或 Postman 确认该批次接口已可用
- 若后端某接口暂未就绪，前端可用 **Mock 数据**（在 `api/` 层返回硬编码数据）先行开发 UI 和交互逻辑
