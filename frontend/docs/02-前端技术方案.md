# 02 — 前端技术方案建议

> **目的**：在 React 18 + TypeScript + Vite 已有骨架之上，给出轻量级前端技术选型建议。每条选择附带理由，控制在课程设计可控复杂度内。

---

## 选型总览

| 层面 | 推荐方案 | 一句话理由 |
|:---|:---|:---|
| UI 样式 | **Tailwind CSS v3** | 原子化CSS，生产包极小，开发效率高 |
| 无样式组件 | **Headless UI** | 仅提供可访问性逻辑，样式完全自控，总包 <10KB |
| 路由 | **React Router v6** | 生态标配，支持嵌套路由与路由守卫 |
| HTTP 客户端 | **Axios** | 拦截器机制天然适配JWT无感注入与401统一处理 |
| 服务端状态 | **TanStack Query v5** | 内置缓存/乐观更新/重试，完美覆盖"点赞收藏不卡顿"需求 |
| 表单校验 | **React Hook Form + Zod** | 性能最优（非受控模式），schema校验自动推导TS类型 |
| 图片压缩 | **browser-image-compression** | 上传前在浏览器端压缩到合理尺寸，节省带宽与存储 |
| Toast 提示 | **react-hot-toast** | 5KB，API极简，支持Promise自动状态切换 |
| 图标 | **Lucide React** | tree-shakeable，按需引入，图标设计现代统一 |
| 全局状态 | **Zustand** | 比Context API更少模板代码，无Provider嵌套，~1KB |

---

## 逐项详细说明

### 1. UI 样式方案 —— Tailwind CSS v3

**推荐理由**：

- **体积优势**：构建时自动摇树（purge），只保留你用到的 class，最终 CSS 通常 <10KB。远轻于 Ant Design（~500KB+）或 MUI
- **开发效率**：样式与组件同文件（`className="flex gap-2 rounded-lg shadow"`），无需在 CSS 文件和 TSX 之间来回切换
- **响应式友好**：内置 `sm:` / `md:` / `lg:` 断点，一行 class 即可适配移动端
- **对学习 React 的助益**：Tailwind 的"组件化思维"与 React 的组件化天然契合 —— 抽取一个 `ShopCard` 组件时，样式随之封装

**备选方案**：如果你试用 Tailwind 后觉得 class 太长影响阅读，可退回 **CSS Modules**（Vite 原生支持，零配置，`*.module.css` 自动作用域隔离）。两者可混用，不冲突。

**不推荐 Ant Design / MUI 的原因**：
- 体积庞大（tree-shaking 后仍常残留 100KB+）
- 定制主题需深入其 Design Token 体系，学习成本高
- 组件自带样式容易与自定义需求冲突，最终"既要改样式又要读源码"
- 你的项目页面数量有限（~12个），手写或Tailwind手写的总代码量不高于配置一个重型组件库

---

### 2. 无样式组件库 —— Headless UI

**推荐理由**：

- **可访问性内置**：Dialog（模态框）、Listbox（下拉选择）、Menu（菜单）、Tabs 等组件自动处理焦点管理、键盘导航、ARIA属性
- **样式完全自控**：组件只提供状态与事件，渲染什么标签、长什么样完全由你决定，与 Tailwind 完美配合
- **极轻量**：tree-shaking 后实际引入的代码 <10KB

**适用场景**：
- 登录/注册引导模态框（游客拦截）
- 品类/区域下拉筛选
- 举报原因选择下拉
- 合并店铺确认弹窗（管理员）

---

### 3. HTTP 客户端 —— Axios

**推荐理由**：

- **请求拦截器**：在 `request interceptor` 中统一注入 JWT Token（从 localStorage 读取），业务代码无需手动传 Token
- **响应拦截器**：在 `response interceptor` 中统一处理 401（Token过期 → 清除登录态 → 跳转登录页），业务代码只需处理正常响应
- **请求/响应类型推导**：结合 TypeScript 泛型，`axios.get<Shop[]>(...)` 直接获得类型安全的 `response.data`
- **体积**：约 13KB gzipped，可接受

**替代方案**：原生 `fetch` + 自定义 wrapper。优点是零依赖，缺点是需要手动封装拦截器、超时、请求取消等逻辑。如果想把学习焦点集中在 React 本身而非工具函数，直接用 Axios。

---

### 4. 服务端状态管理 —— TanStack Query v5

**推荐理由**：

- **直接解决"乐观更新"需求**：`useMutation` 提供 `onMutate` 回调（乐观修改缓存）和 `onError` 回调（回滚缓存），完美实现"点击收藏→图标立刻变色→请求失败则恢复"的体验
- **缓存策略内置**：同一店铺列表在不同页面（首页、发现页、收藏夹）共享缓存，避免重复请求
- **自动处理加载/错误状态**：`isLoading`、`isError`、`data` 由 hook 统一管理，减少组件内的 `useState` + `useEffect` 样板代码
- **体积**：约 13KB gzipped，物超所值
- **行业标准**：React 生态中服务端状态管理的实际标准，值得在课设中学习

**客户端全局状态**（Zustand）：TanStack Query 管理"来自服务端的数据"，而 Zustand 管理"纯客户端状态"——
- 当前登录用户信息
- 搜索历史（本地存储）
- UI 状态（侧边栏展开/收起、当前页面标题等）

两者职责清晰，互不冲突。

---

### 5. 表单校验 —— React Hook Form + Zod

**推荐理由**：

- **React Hook Form**：使用非受控模式（ref 注册），输入变更不触发组件重渲染，性能远超受控表单
- **Zod**：TypeScript-first 的 schema 校验库，定义一次 schema 即可同时获得运行时校验 + TypeScript 类型推导
- **组合方式**：`@hookform/resolvers` 桥接 Zod schema，校验规则与组件解耦，方便复用

**适用场景**（你的项目中至少有以下表单）：
- 登录表单（邮箱/手机号 + 密码）
- 注册表单（邮箱/手机号 + 密码 + 昵称）
- 上传店铺表单（名称 + 区域 + 品类）
- 评论发布表单（文字 + 可选图片）
- 问题/回答表单
- 个人信息编辑表单
- 举报表单（选择原因）
- 管理员公告发布表单

---

### 6. 图片上传处理策略

**三个阶段**：

| 阶段 | 处理方式 | 工具/方案 |
|:---|:---|:---|
| **上传前** | 前端校验格式（jpg/png）+ 大小（≤5MB）+ 压缩 | `browser-image-compression` 将图片压缩至宽度 ≤1920px、质量 0.8 |
| **上传中** | 显示进度条或骨架屏，防止用户误以为"卡死了" | Axios `onUploadProgress` 回调 + 进度条组件 |
| **上传后** | 后端返回图片URL，前端直接展示；上传失败 Toast 提示重试 | 后端存储至 `/static/uploads/`，Nginx 直接提供静态资源 |

**图片压缩是必选项**：大二学生用手机拍照动辄 5-10MB，不压缩直接上传会严重拖慢发布体验，且白占服务器磁盘。`browser-image-compression` 约 8KB，压缩一张 5MB 照片到 ~200KB 通常在 2 秒内完成。

---

### 7. 其他推荐

| 工具 | 用途 | 包大小 |
|:---|:---|:---|
| **react-hot-toast** | Toast 轻提示（收藏成功/发布成功/网络异常等） | ~5KB |
| **Lucide React** | 图标库（收藏星、点赞心、搜索、通知铃等），按需引入 | 每个图标 ~1-2KB |
| **Zustand** | 客户端全局状态 | ~1KB |
| **clsx** | 条件拼接 className（如 `clsx('btn', isActive && 'btn--active')`） | <1KB |
