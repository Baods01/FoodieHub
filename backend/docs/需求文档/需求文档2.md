## 店铺合并功能需求说明

### 1. 背景与目标

在用户自主上传店铺的过程中，可能出现不同用户对同一实体店铺使用了不同名称（如“麦当劳（望京店）”与“麦当劳望京餐厅”），导致数据库中产生重复的店铺实体。为维护数据规范、提升用户体验，需要为管理员提供**店铺合并**功能，将同一现实店铺的多个记录合并为一个统一的店铺记录，同时保留所有关联内容。

### 2. 前置条件

- 仅**管理员**角色（`role=1`）可执行合并操作。
- 参与合并的店铺必须满足：
  - 均为**已通过审核**且**未被软删除**的店铺（`status='approved'` 且 `is_active=1`）。
  - 至少选择 **2 个**店铺，最多建议不超过 5 个（避免数据冲突过于复杂）。

### 3. 合并操作流程

#### 3.1 选择目标店铺

管理员在后台店铺管理列表中，勾选需要合并的店铺记录，点击“合并”按钮。

#### 3.2 确认合并信息

系统弹窗展示参与合并的店铺列表，要求管理员指定：

- **主店铺名称**：从所有合并店铺的 `name` 字段中选择一个作为合并后的店铺名称。
- **店铺描述**：从所有合并店铺的 `description` 字段中选择一个保留（管理员可在此处编辑微调，但不可手动输入全新内容）。
- 确认合并后，系统将执行合并逻辑。

#### 3.3 执行合并事务

合并过程应在数据库**事务**中完成，确保原子性：要么全部成功，要么全部回滚。

### 4. 数据合并规则（详细）

设参与合并的店铺集合为 `S = {s1, s2, ..., sn}`，合并后的新店铺为 `s_new`。

| 数据类型               | 合并策略                                                                                                                                                                                                                                                                                                                               |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **基本信息**     | -`name`：管理员选择的名称。`<br>`- `description`：管理员选择的描述。`<br>`- `is_oncampus`：若所有店铺的 `is_oncampus` 值相同则保留，否则设置为 `NULL`（表示待管理员修正）。`<br>`- `address`、`phone`、`latitude`、`longitude`：优先取非空值，若多个店铺有非空值且内容不同，则拼接（用 `;`分隔）并记录日志。 |
| **别名**         | 除主名称外，其他店铺的 `name` 存入新字段 `aliases`（TEXT或JSON），格式如 `["原店名A", "原店名B"]`。**需在 `shops` 表中增加 `aliases` 字段**。                                                                                                                                                                          |
| **浏览量**       | `view_count = SUM(各店铺的 view_count)`                                                                                                                                                                                                                                                                                              |
| **收藏量**       | 重新计算：`favorite_count = COUNT(DISTINCT user_id)`，其中 `user_id` 取自所有合并店铺的 `favorites` 表中记录（同一用户收藏多个被合并店铺只计一次）。`<br>`同时将那些用户的收藏记录**迁移**到 `s_new`（删除原记录，插入新记录）。                                                                                       |
| **评论**         | 将所有 `comments` 表中 `shop_id` 指向 `s_i` 的记录，更新 `shop_id` 为 `s_new.id`。                                                                                                                                                                                                                                           |
| **评分**         | - 每个用户的评分取**最高分**：若同一用户对多个被合并店铺评过分，只保留其给出的最高分，作为该用户对 `s_new` 的评分。`<br>`- 迁移后重新计算 `average_rating` 和 `total_ratings`。`<br>`- 原评分记录中未被保留的（即用户较低分）应**删除**。                                                                        |
| **菜单项**       | 所有 `menu_items` 中 `shop_id` 指向 `s_i` 的记录，更新为 `s_new.id`。`<br>`若出现完全相同的菜品名（`name` 相同且 `price` 相同），只保留一条（可保留 `description` 较长的）。                                                                                                                                           |
| **图片**         | 所有 `images` 中 `entity_type='shop'` 且 `entity_id` 指向 `s_i` 的记录，更新 `entity_id` 为 `s_new.id`。`<br>`若图片重复（相同 URL），去重。                                                                                                                                                                             |
| **点餐方式**     | 收集所有被合并店铺的 `dining_method_id`（通过 `shop_dining_methods` 中间表），去重后关联到 `s_new`。                                                                                                                                                                                                                             |
| **品类**         | 收集所有被合并店铺的 `category_id`（通过 `shop_categories` 中间表），去重后关联到 `s_new`。                                                                                                                                                                                                                                      |
| **动态**         | 所有 `activities` 中 `target_type='shop'` 且 `target_id` 指向 `s_i` 的记录，更新 `target_id` 为 `s_new.id`。                                                                                                                                                                                                               |
| **行为日志**     | 所有 `user_behavior_logs` 中 `target_type='shop'` 且 `target_id` 指向 `s_i` 的记录，更新 `target_id` 为 `s_new.id`。                                                                                                                                                                                                       |
| **店铺修改申请** | 所有 `shop_edit_requests` 中 `shop_id` 指向 `s_i` 且状态为 `pending` 的记录，更新 `shop_id` 为 `s_new.id`；已处理（`approved`/`rejected`）的记录保持不变。                                                                                                                                                             |
| **被合并店铺**   | 将 `s_i`（i=1..n）的 `is_active` 设置为 `FALSE`（软删除），并记录 `merged_into = s_new.id`（需在 `shops` 表增加 `merged_into` 字段，指向新店铺）。                                                                                                                                                                         |

### 5. 新增字段

为支持合并功能，需在 `shops` 表中增加以下字段：

| 字段名      | 类型 | 属性                           | 说明                                      |
| ----------- | ---- | ------------------------------ | ----------------------------------------- |
| aliases     | JSON | 可选                           | 存储别名列表，例如 `["老店名", "别名"]` |
| merged_into | INT  | 外键引用 `shops(id)`，可为空 | 若店铺被合并，该字段指向合并后的新店铺ID  |

### 6. 异常处理与日志

- 合并过程中若发生任何异常（如数据库唯一键冲突、外键错误），事务回滚，并向管理员提示“合并失败，请检查数据或联系技术支持”。
- 每次合并操作需记录**管理员操作日志**（可复用 `user_behavior_logs`，`behavior_type='merge_shops'`，`extra_data` 存储参与合并的店铺ID列表和选择的主名称）。

### 7. 合并后的后续处理

- **前端展示**：访问原店铺 URL 时，应自动 301/302 重定向到合并后店铺的详情页，并提示“该店铺已与【新店铺名】合并”。
- **搜索引擎索引**：建议在合并时更新站点地图，通知搜索引擎原店铺已变更。
- **数据一致性**：合并后应触发异步任务重新计算新店铺的热度分数（如 `weekly_hot_score`）。

### 8. 权限与安全

- 仅超级管理员或具有“店铺管理”权限的管理员可执行合并。
- 操作前需二次确认，防止误操作。
- 合并操作不可逆（除非从备份恢复），应提示管理员谨慎操作。
