## 1. 项目概述

### 1.1 项目名称

FoodieHub —— 周边食品店铺点评平台

### 1.2 项目背景

在信息繁杂的点评平台中，用户难以获得真实、可靠的小范围店铺推荐。FoodieHub 致力于为内部小群体（如同事、同学、社区成员）提供一个信息真实、互动紧密的店铺分享与点评平台。所有店铺均由用户自主上传，确保平台上出现的每家店铺都具有真实的分享价值。

### 1.3 目标用户

- **普通用户**：可以在平台上浏览、搜索店铺，上传新店铺，对已有店铺进行评分、评论、提问、回答，收藏店铺，查看个人动态和消息通知。
- **管理员**：负责审核店铺信息修改申请，管理用户（禁用/启用），管理店铺（下架/恢复），发布公告，以及**执行店铺合并操作**。

### 1.4 核心价值

- **信息真实**：用户自主上传，内容经管理员审核，杜绝虚假或重复店铺。
- **互动紧密**：评分、评论、问答等多维度交互，形成社区氛围。
- **信息流通**：个人动态、消息通知及时同步用户参与行为，促进二次互动。
- **数据规范**：通过店铺合并功能，消除重复店铺记录，维护数据库整洁。

## 2. 功能需求

### 2.1 用户模块

#### 2.1.1 注册与登录

- 用户通过手机号或邮箱注册，设置用户名和密码。
- 支持密码加密存储。
- 支持“记住我”功能，保持登录状态。

#### 2.1.2 个人信息管理

- 用户可修改头像、昵称、个人简介。
- 可查看自己的收藏夹、发布的评论/问题/回答、上传的店铺。
- 支持账号注销（软删除）。

#### 2.1.3 用户角色

- 普通用户（role = 0）
- 管理员（role = 1）

---

### 2.2 店铺模块

#### 2.2.1 店铺上传

- 用户可上传新店铺，需填写：
  - 店铺名称（必填）
  - 店铺描述（可选）
  - 店铺类别（可多选但必选一项）
  - 点餐方式（多选，如堂食、外卖、自取，但必选一项）
  - 校内/校外标识（必选其一）
  - 图片（可选，可上传多张）
- 上传后直接在前台展示。
- 若店铺名称已存在，提示用户“该店铺已存在”，并弹出跳转链接供其跳转。

#### 2.2.2 店铺查看

- 店铺详情页展示：
  - 基础信息（名称、描述、类别、点餐方式、校内校外）
  - 图片轮播（支持上传）
  - 平均评分（1-5星）
  - 菜单列表（菜品名称、价格、描述）
  - 问答区（提问/回答）
  - 用户评论列表（支持回复）

#### 2.2.3 店铺交互

- **评分**：用户可对店铺打1-5星，每个用户仅能评一次，可修改评分。
- **评论**：用户可对店铺发表文字评论，支持回复其他评论（形成多级评论）。
- **问答**：用户可对店铺提问，其他用户可回答（形成多级问答）。
- **菜单**：用户可查看店铺菜单（菜品名称、价格、描述）。
- **图片**：用户可查看店铺图片，并可上传补充图片。

#### 2.2.4 店铺信息修改

- 任何用户均可提交店铺基础信息修改申请（如类别、点餐方式等），修改内容需管理员审核。
- 申请记录保留，便于追溯。

#### 2.2.5 店铺管理（管理员）

- 审核店铺信息修改申请：通过/拒绝。
- 下架/恢复店铺（软删除），下架后前台不可见。
- 删除违规图片或评论。
- **店铺合并**（详见第 2.7 节）

#### 2.2.6 店铺合并（新增功能）

##### 2.2.6.1 背景与目的

在用户自主上传店铺的过程中，可能出现不同用户对同一实体店铺使用了不同名称（如“麦当劳（望京店）”与“麦当劳望京餐厅”），导致数据库中产生重复的店铺实体。为维护数据规范、提升用户体验，管理员可执行**店铺合并**操作，将多个重复店铺合并为一个统一店铺，并保留所有关联内容。

##### 2.2.6.2 前置条件

- 仅管理员角色（`role=1`）可执行合并操作。
- 参与合并的店铺必须满足：
  - 均为**已通过审核**且**未被软删除**的店铺（`status='approved'` 且 `is_active=1`）。
  - 至少选择 **2 个**店铺，最多建议不超过 5 个（避免数据冲突过于复杂）。

##### 2.2.6.3 合并操作流程

1. **选择目标店铺**：管理员在后台店铺管理列表中，勾选需要合并的店铺记录，点击“合并”按钮。
2. **确认合并信息**：系统弹窗展示参与合并的店铺列表，要求管理员指定：
   - **主店铺名称**：从所有合并店铺的 `name` 字段中选择一个作为合并后的店铺名称。
   - **店铺描述**：从所有合并店铺的 `description` 字段中选择一个保留（管理员可在此处编辑微调，但不可手动输入全新内容）。
3. **执行合并事务**：合并过程在数据库事务中完成，确保原子性（全部成功或全部回滚）。

##### 2.2.6.4 数据合并规则

设参与合并的店铺集合为 `S = {s1, s2, ..., sn}`，合并后的新店铺为 `s_new`。

| 数据类型               | 合并策略                                                                                                                                                                                                                                                                                                                            |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **基本信息**     | -`name`：管理员选择的名称。`<br>`- `description`：管理员选择的描述。`<br>`- `is_oncampus`：若所有店铺的 `is_oncampus` 值相同则保留，否则设置为 `NULL`（待管理员修正）。`<br>`- `address`、`phone`、`latitude`、`longitude`：优先取非空值，若多个店铺有非空值且内容不同，则拼接（用 `;` 分隔）并记录日志。 |
| **别名**         | 除主名称外，其他店铺的 `name` 存入 `aliases` 字段（JSON 格式），如 `["原店名A", "原店名B"]`。                                                                                                                                                                                                                                 |
| **浏览量**       | `view_count = SUM(各店铺的 view_count)`                                                                                                                                                                                                                                                                                           |
| **收藏量**       | 重新计算：`favorite_count = COUNT(DISTINCT user_id)`，其中 `user_id` 取自所有合并店铺的 `favorites` 表中记录（同一用户收藏多个被合并店铺只计一次）。同时将这些用户的收藏记录迁移到 `s_new`（删除原记录，插入新记录）。                                                                                                      |
| **评论**         | 将所有 `comments` 表中 `shop_id` 指向 `s_i` 的记录，更新 `shop_id` 为 `s_new.id`。                                                                                                                                                                                                                                        |
| **评分**         | - 每个用户的评分取最高分：若同一用户对多个被合并店铺评过分，只保留其给出的最高分，作为该用户对 `s_new` 的评分。`<br>`- 迁移后重新计算 `average_rating` 和 `total_ratings`。`<br>`- 原评分记录中未被保留的（即用户较低分）应删除。                                                                                         |
| **菜单项**       | 所有 `menu_items` 中 `shop_id` 指向 `s_i` 的记录，更新为 `s_new.id`。若出现完全相同的菜品名（`name` 相同且 `price` 相同），只保留一条（保留 `description` 较长的）。                                                                                                                                                  |
| **图片**         | 所有 `images` 中 `entity_type='shop'` 且 `entity_id` 指向 `s_i` 的记录，更新 `entity_id` 为 `s_new.id`。若图片重复（相同 URL），去重。                                                                                                                                                                                  |
| **点餐方式**     | 收集所有被合并店铺的 `dining_method_id`（通过 `shop_dining_methods` 中间表），去重后关联到 `s_new`。                                                                                                                                                                                                                          |
| **品类**         | 收集所有被合并店铺的 `category_id`（通过 `shop_categories` 中间表），去重后关联到 `s_new`。                                                                                                                                                                                                                                   |
| **动态**         | 所有 `activities` 中 `target_type='shop'` 且 `target_id` 指向 `s_i` 的记录，更新 `target_id` 为 `s_new.id`。                                                                                                                                                                                                            |
| **行为日志**     | 所有 `user_behavior_logs` 中 `target_type='shop'` 且 `target_id` 指向 `s_i` 的记录，更新 `target_id` 为 `s_new.id`。                                                                                                                                                                                                    |
| **店铺修改申请** | 所有 `shop_edit_requests` 中 `shop_id` 指向 `s_i` 且状态为 `pending` 的记录，更新 `shop_id` 为 `s_new.id`；已处理（`approved`/`rejected`）的记录保持不变。                                                                                                                                                          |
| **被合并店铺**   | 将 `s_i`（i=1..n）的 `is_active` 设置为 `FALSE`（软删除），并记录 `merged_into = s_new.id`。                                                                                                                                                                                                                                |

##### 2.2.6.5 新增字段

为支持合并功能，需在 `shops` 表中增加以下字段：

| 字段名      | 类型 | 属性                           | 说明                                      |
| ----------- | ---- | ------------------------------ | ----------------------------------------- |
| aliases     | JSON | 可选                           | 存储别名列表，例如 `["老店名", "别名"]` |
| merged_into | INT  | 外键引用 `shops(id)`，可为空 | 若店铺被合并，该字段指向合并后的新店铺ID  |

##### 2.2.6.6 异常处理与日志

- 合并过程中若发生任何异常（如数据库唯一键冲突、外键错误），事务回滚，并向管理员提示“合并失败，请检查数据或联系技术支持”。
- 每次合并操作需记录管理员操作日志（可复用 `user_behavior_logs`，`behavior_type='merge_shops'`，`extra_data` 存储参与合并的店铺ID列表和选择的主名称）。

##### 2.2.6.7 合并后的后续处理

- **前端展示**：访问原店铺 URL 时，应自动 301/302 重定向到合并后店铺的详情页，并提示“该店铺已与【新店铺名】合并”。
- **搜索引擎索引**：建议在合并时更新站点地图，通知搜索引擎原店铺已变更。
- **数据一致性**：合并后应触发异步任务重新计算新店铺的热度分数（如 `weekly_hot_score`）。

##### 2.2.6.8 权限与安全

- 仅超级管理员或具有“店铺管理”权限的管理员可执行合并。
- 操作前需二次确认，防止误操作。
- 合并操作不可逆（除非从备份恢复），应提示管理员谨慎操作。

## 3. 业务规则

### 3.1 店铺上传规则

- 店铺名称不能与已存在的店铺重复（忽略大小写）。
- 上传的店铺图片无需审核，用户上传后直接展示，管理员可事后删除违规图片。
- 店铺信息修改申请提交后，原店铺信息保持不变，直至申请通过。

### 3.2 评论与问答规则

- 评论支持无限级回复，除第一级评论，再往下的评论都处在同一缩进（回复开头@回复的上一级评论的用户即可）。
- 用户可删除自己的评论/提问/回答（软删除）。
- 管理员可删除任何违规评论/提问/回答。

### 3.3 评分规则

- 每个用户对同一店铺只能评一次，允许修改评分，修改后更新平均评分。
- 店铺平均评分 = 所有评分的总分 / 评分人数（保留一位小数）。

### 3.4 动态生成规则

- 当用户执行以下操作时，自动生成一条动态记录：
  - 上传店铺并通过审核
  - 评分
  - 评论
  - 点赞
  - 回答问题
  - 收藏店铺
- 动态内容应包含摘要，例如“用户A 评论了 店铺B：好评！”。

### 3.5 消息触发规则

- 当用户评论被回复时，回复者自动向被回复者发送消息。
- 当用户提问被回答时，回答者自动向提问者发送消息。
- 管理员发布公告时，为所有用户生成一条系统消息。
- 店铺审核结果或修改申请结果，向申请者发送消息。

### 3.6 店铺合并规则（新增）

- 合并后的店铺视为全新实体，原店铺软删除并记录合并关系。
- 合并时需处理数据冲突（如评分取最高分、别名存储、图片去重等）。
- 合并操作不可逆，需管理员二次确认。

## 4. 非功能需求

### 4.1 性能要求

- 页面加载时间 ≤ 2秒（在标准网络环境下）。
- 支持并发用户数 ≥ 1000。
- 店铺合并操作应在合理时间内完成（例如合并 5 个店铺不超过 3 秒）。

### 4.2 安全性

- 用户密码使用 bcrypt 或类似算法加密存储。
- 所有用户输入需进行 XSS 过滤，防止脚本注入。
- 接口需做权限校验，防止越权操作（如普通用户修改他人数据）。
- 店铺合并接口仅限管理员访问，需记录操作日志。

### 4.3 可扩展性

- 图片存储使用静态文件存储方式，数据库只存储相对路径。
- 数据库设计支持软删除（`is_active` 字段），便于数据恢复。
- 店铺合并规则应易于扩展（例如未来支持更多实体的合并）。

### 4.4 易用性

- 界面简洁清晰。
- 图片懒加载，提升首屏体验。
- 管理员合并操作界面应提供清晰的店铺信息对比和冲突提示。

---

## 5. 数据要求（简要）

基于数据模型设计，需满足以下核心约束：

- 每个用户对同一店铺仅有一条评分记录（唯一约束）。
- 每个用户对同一店铺仅有一条收藏记录（唯一约束）。
- 评论表支持自关联（`parent_id`、`root_id`），便于树形结构查询。
- 图片表采用多态关联（`entity_type` + `entity_id`），可关联店铺、菜单、评论等实体。
- 行为日志表记录所有关键操作，用于后续热度分析。
- 店铺表需增加 `aliases` 和 `merged_into` 字段以支持合并功能。
