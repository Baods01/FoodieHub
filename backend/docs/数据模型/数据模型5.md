## 基础模型

### BaseModel（抽象基类）

所有业务模型继承自 `BaseModel`，该基类组合了 `TimestampMixin` 和 `SoftDeleteMixin`。

| 字段名 | 类型 | 约束 | 描述 |
|--------|------|------|------|
| `created_at` | DATETIME | AUTO_NOW_ADD | 创建时间 |
| `updated_at` | DATETIME | AUTO_NOW | 最后更新时间 |
| `is_active` | BOOLEAN | DEFAULT TRUE | 是否启用（软删除标记） |

## 用户模块

### 1. users（用户表）

系统用户信息表，存储注册用户的基本资料和权限信息。

| 字段名 | 类型 | 约束 | 默认值 | 描述 |
|--------|------|------|--------|------|
| `id` | INT | PK | 自增 | 用户唯一标识 |
| `username` | VARCHAR(50) | UNIQUE, NOT NULL | - | 登录用户名 |
| `password` | VARCHAR(255) | NOT NULL | - | 加密后的密码 |
| `phone` | VARCHAR(20) | NOT NULL | - | 手机号 |
| `email` | VARCHAR(100) | NOT NULL | - | 电子邮箱，用于通知 |
| `avatar` | VARCHAR(255) | NULL | - | 头像图片URL |
| `bio` | TEXT | NULL | - | 个人简介 |
| `role` | INT | NOT NULL | 0 | 角色：0=普通用户，1=管理员 |
| `created_at` | DATETIME | AUTO_NOW_ADD | - | 创建时间 |
| `updated_at` | DATETIME | AUTO_NOW | - | 最后更新时间 |
| `is_active` | BOOLEAN | DEFAULT TRUE | TRUE | 是否启用（软删除） |

**关联关系：**
- 一对多 → `activities`（用户动态）
- 一对多 → `favorites`（用户收藏）
- 一对多 → `messages`（接收/发送的消息）
- 一对多 → `ratings`（用户评分）
- 一对多 → `comments`（用户评论）
- 一对多 → `comments_likes`（用户点赞）
- 一对多 → `shop_edit_requests`（提交的申请）
- 一对多 → `user_behavior_logs`（行为日志）

---

### 2. activities（动态表）

记录用户在系统中的各类操作动态，用于时间线展示。

| 字段名 | 类型 | 约束 | 默认值 | 描述 |
|--------|------|------|--------|------|
| `id` | INT | PK | 自增 | 动态唯一标识 |
| `user_id` | INT | FK → users.id, NOT NULL | - | 产生动态的用户 |
| `type` | VARCHAR(50) | NOT NULL | - | 动态类型：rating、comment、question、answer、favorite、add_shop 等 |
| `target_id` | INT | NOT NULL | - | 关联目标ID（如评论ID、店铺ID等） |
| `target_type` | VARCHAR(50) | NOT NULL | - | 目标实体类型，便于前端跳转 |
| `content` | VARCHAR(255) | NULL | - | 动态摘要，如"评论了店铺XX" |
| `created_at` | DATETIME | AUTO_NOW_ADD | - | 创建时间 |
| `updated_at` | DATETIME | AUTO_NOW | - | 最后更新时间 |
| `is_active` | BOOLEAN | DEFAULT TRUE | TRUE | 是否启用（软删除） |

**索引：**
- `idx_activities_user_created` (`user_id`, `created_at`) - 用户时间线查询

**设计说明：** 采用多态关联（`target_id` + `target_type`），可灵活关联到不同实体。

---

### 3. favorites（收藏表）

记录用户收藏的店铺信息。

| 字段名 | 类型 | 约束 | 默认值 | 描述 |
|--------|------|------|--------|------|
| `id` | INT | PK | 自增 | 收藏唯一标识 |
| `user_id` | INT | FK → users.id, NOT NULL | - | 收藏用户 |
| `shop_id` | INT | FK → shops.id, NOT NULL | - | 被收藏店铺 |
| `sort_order` | INT | NOT NULL | 0 | 排序序号，数值越小越靠前 |
| `created_at` | DATETIME | AUTO_NOW_ADD | - | 创建时间 |
| `updated_at` | DATETIME | AUTO_NOW | - | 最后更新时间 |
| `is_active` | BOOLEAN | DEFAULT TRUE | TRUE | 是否启用（软删除） |

**唯一约束：** `(user_id, shop_id)` - 防止重复收藏同一店铺

---

### 4. messages（消息表）

系统消息和用户通知表。

| 字段名 | 类型 | 约束 | 默认值 | 描述 |
|--------|------|------|--------|------|
| `id` | INT | PK | 自增 | 消息唯一标识 |
| `recipient_id` | INT | FK → users.id, NOT NULL | - | 接收用户 |
| `sender_id` | INT | FK → users.id, NULL | - | 发送方（系统消息时为空） |
| `type` | VARCHAR(50) | NOT NULL | - | 类型：announcement、reply_comment、reply_answer 等 |
| `title` | VARCHAR(100) | NULL | - | 消息标题 |
| `content` | TEXT | NOT NULL | - | 消息内容 |
| `related_entity_type` | VARCHAR(50) | NULL | - | 关联实体类型（如 comment、answer） |
| `related_entity_id` | INT | NULL | - | 关联实体ID，用于跳转 |
| `created_at` | DATETIME | AUTO_NOW_ADD | - | 创建时间 |
| `updated_at` | DATETIME | AUTO_NOW | - | 最后更新时间 |
| `is_active` | BOOLEAN | DEFAULT TRUE | TRUE | 是否启用（软删除） |

**索引：**
- `idx_messages_recipient_created` (`recipient_id`, `created_at`) - 消息列表查询

## 店铺模块

### 5. shops（店铺表）

核心业务表，存储店铺基本信息。

| 字段名 | 类型 | 约束 | 默认值 | 描述 |
|--------|------|------|--------|------|
| `id` | INT | PK | 自增 | 店铺唯一标识 |
| `name` | VARCHAR(100) | NOT NULL | - | 店铺名称 |
| `description` | TEXT | NULL | - | 店铺描述 |
| `view_count` | INT | DEFAULT 0 | 0 | 总浏览量（冗余字段） |
| `favorite_count` | INT | DEFAULT 0 | 0 | 收藏数（冗余字段） |
| `comment_count` | INT | DEFAULT 0 | 0 | 讨论数（冗余字段） |
| `average_rating` | DECIMAL(2,1) | DEFAULT 0.0 | 0.0 | 平均评分（冗余字段） |
| `aliases` | JSON | NULL | - | 存储别名列表，例如 ["老店名", "别名"] |
| `merged_into_id` | INT | FK → shops.id, NULL | - | 合并后新店铺 |
| `created_at` | DATETIME | AUTO_NOW_ADD | - | 创建时间 |
| `updated_at` | DATETIME | AUTO_NOW | - | 最后更新时间 |
| `is_active` | BOOLEAN | DEFAULT TRUE | TRUE | 是否启用（软删除） |

**关联关系：**
- 一对多 → `menu_items`（菜单项）
- 一对多 → `ratings`（评分）
- 一对多 → `comments`（评论）
- 多对多 → `dict_data`（通过 `shop_dict_rel` 关联）
- 自引用 → `merged_into`（店铺合并关系）

---

### 6. menu_items（菜单项表）

店铺菜品/菜单项信息表。

| 字段名 | 类型 | 约束 | 默认值 | 描述 |
|--------|------|------|--------|------|
| `id` | INT | PK | 自增 | 菜单项唯一标识 |
| `shop_id` | INT | FK → shops.id, NOT NULL | - | 所属店铺 |
| `name` | VARCHAR(100) | NOT NULL | - | 菜品名称 |
| `price` | DECIMAL(10,2) | NULL | - | 价格 |
| `description` | TEXT | NULL | - | 菜品描述 |
| `created_at` | DATETIME | AUTO_NOW_ADD | - | 创建时间 |
| `updated_at` | DATETIME | AUTO_NOW | - | 最后更新时间 |
| `is_active` | BOOLEAN | DEFAULT TRUE | TRUE | 是否启用（软删除） |

---

### 7. ratings（评分表）

用户对店铺的评分记录。

| 字段名 | 类型 | 约束 | 默认值 | 描述 |
|--------|------|------|--------|------|
| `id` | INT | PK | 自增 | 评分唯一标识 |
| `user_id` | INT | FK → users.id, NOT NULL | - | 评分用户 |
| `shop_id` | INT | FK → shops.id, NOT NULL | - | 被评店铺 |
| `score` | INT | NOT NULL | - | 评分值（建议约束 1-5） |
| `created_at` | DATETIME | AUTO_NOW_ADD | - | 创建时间 |
| `updated_at` | DATETIME | AUTO_NOW | - | 最后更新时间 |
| `is_active` | BOOLEAN | DEFAULT TRUE | TRUE | 是否启用（软删除） |

**唯一约束：** `(user_id, shop_id)` - 确保每用户对每店铺只能评分一次

---

### 8. comments（评论表）

店铺评论及回复表，支持多级评论结构。

| 字段名 | 类型 | 约束 | 默认值 | 描述 |
|--------|------|------|--------|------|
| `id` | BIGINT | PK | 自增 | 评论唯一标识 |
| `shop_id` | INT | FK → shops.id, NOT NULL | - | 关联的店铺 |
| `user_id` | INT | FK → users.id, NOT NULL | - | 发表评论的用户 |
| `type` | VARCHAR(20) | DEFAULT 'comment' | 'comment' | 评论类型：question 或 comment |
| `parent_id` | BIGINT | FK → comments.id, NULL | - | 父评论，NULL表示一级评论 |
| `root_id` | BIGINT | FK → comments.id, NULL | - | 根评论（所属的一级评论） |
| `content` | TEXT | NOT NULL | - | 评论内容，支持富文本或纯文本 |
| `like_count` | INT | NOT NULL | 0 | 点赞数（冗余字段） |
| `reply_count` | INT | NOT NULL | 0 | 直接子回复数量（冗余字段） |
| `created_at` | DATETIME | AUTO_NOW_ADD | - | 创建时间 |
| `updated_at` | DATETIME | AUTO_NOW | - | 最后更新时间 |
| `is_active` | BOOLEAN | DEFAULT TRUE | TRUE | 是否启用（软删除） |

**索引：**
- `idx_comments_shop_root_created` (`shop_id`, `root_id`, `created_at`) - 评论列表查询

**设计说明：** 采用 `parent_id` + `root_id` 双指针结构，便于高效查询评论树和聚合统计。

---

### 9. comments_likes（评论点赞表）

记录用户对评论的点赞行为。

| 字段名 | 类型 | 约束 | 默认值 | 描述 |
|--------|------|------|--------|------|
| `id` | INT | PK | 自增 | 点赞唯一标识 |
| `user_id` | INT | FK → users.id, NOT NULL | - | 点赞用户 |
| `comment_id` | BIGINT | FK → comments.id, NOT NULL | - | 被点赞的评论 |
| `created_at` | DATETIME | AUTO_NOW_ADD | - | 创建时间 |
| `updated_at` | DATETIME | AUTO_NOW | - | 最后更新时间 |
| `is_active` | BOOLEAN | DEFAULT TRUE | TRUE | 是否启用（软删除） |

**唯一约束：** `(user_id, comment_id)` - 防止重复点赞

## 字典模块

### 10. dict_types（字典类型表）

字典分类定义表，用于管理系统中的各类字典数据。

| 字段名 | 类型 | 约束 | 默认值 | 描述 |
|--------|------|------|--------|------|
| `id` | INT | PK | 自增 | 唯一标识 |
| `code` | VARCHAR(50) | UNIQUE, NOT NULL | - | 类型编码，如 category、dining_method、location_type |
| `name` | VARCHAR(50) | UNIQUE, NOT NULL | - | 类型名称，如店铺类别、点餐方式、位置类型 |
| `target_table` | VARCHAR(50) | NOT NULL | - | 目标表名，标识该字典类型属于哪个表 |
| `description` | VARCHAR(255) | NULL | - | 类型描述 |
| `sort_order` | INT | DEFAULT 0 | 0 | 排序顺序（类型间排序） |
| `created_at` | DATETIME | AUTO_NOW_ADD | - | 创建时间 |
| `updated_at` | DATETIME | AUTO_NOW | - | 最后更新时间 |
| `is_active` | BOOLEAN | DEFAULT TRUE | TRUE | 是否启用（软删除） |

---

### 11. dict_data（字典数据表）

具体字典数据项表。

| 字段名 | 类型 | 约束 | 默认值 | 描述 |
|--------|------|------|--------|------|
| `id` | INT | PK | 自增 | 唯一标识 |
| `dict_type_id` | INT | FK → dict_types.id, NOT NULL | - | 所属字典类型 |
| `code` | VARCHAR(50) | NOT NULL | - | 数据编码（同一类型内唯一） |
| `name` | VARCHAR(50) | NOT NULL | - | 显示名称，如中餐 |
| `value` | VARCHAR(255) | NULL | - | 额外值（如保留原始值） |
| `sort_order` | INT | DEFAULT 0 | 0 | 排序顺序（同一类型内） |
| `is_default` | BOOLEAN | DEFAULT FALSE | FALSE | 是否为默认值 |
| `extra` | JSON | NULL | - | 扩展字段（如存储图标、颜色等） |
| `created_at` | DATETIME | AUTO_NOW_ADD | - | 创建时间 |
| `updated_at` | DATETIME | AUTO_NOW | - | 最后更新时间 |
| `is_active` | BOOLEAN | DEFAULT TRUE | TRUE | 是否启用（软删除） |

**唯一约束：** `(dict_type_id, code)` - 防止同一类型下重复编码

---

| `is_active` | BOOLEAN | DEFAULT TRUE | TRUE | 是否启用（软删除） |

## 日志模块

### 13. user_behavior_logs（用户行为日志表）

记录用户在系统中的行为操作，用于数据分析和行为追踪。

| 字段名 | 类型 | 约束 | 默认值 | 描述 |
|--------|------|------|--------|------|
| `id` | BIGINT | PK | 自增 | 日志唯一标识 |
| `user_id` | INT | FK → users.id, NULL | - | 操作用户（未登录用户可为空） |
| `session_id` | VARCHAR(64) | NULL | - | 会话ID，用于追踪未登录用户行为 |
| `behavior_type` | VARCHAR(32) | NOT NULL | - | 行为类型：view_shop、rate_shop、comment、favorite、share 等 |
| `target_type` | VARCHAR(32) | NOT NULL | - | 操作目标类型：shop、menu_item、comment、question 等 |
| `target_id` | INT | NOT NULL | - | 目标对象的ID |
| `ip_address` | VARCHAR(45) | NULL | - | 客户端IP地址 |
| `user_agent` | TEXT | NULL | - | 客户端设备信息 |
| `created_at` | DATETIME | AUTO_NOW_ADD | - | 创建时间 |
| `updated_at` | DATETIME | AUTO_NOW | - | 最后更新时间 |
| `is_active` | BOOLEAN | DEFAULT TRUE | TRUE | 是否启用（软删除） |

**索引：**
- `idx_logs_user_created` (`user_id`, `created_at`) - 用户行为查询按时间范围过滤
- `idx_logs_behavior_created` (`behavior_type`, `created_at`) - 行为统计分析

**设计说明：** 采用多态关联（`target_type` + `target_id`），可记录对任意实体的操作。

## 审核模块

### 14. shop_edit_requests（店铺编辑申请表）

用户提交店铺修改申请，管理员审核通过后生效。

| 字段名 | 类型 | 约束 | 默认值 | 描述 |
|--------|------|------|--------|------|
| `id` | INT | PK | 自增 | 申请唯一标识 |
| `shop_id` | INT | FK → shops.id, NOT NULL | - | 待修改的店铺 |
| `user_id` | INT | FK → users.id, NOT NULL | - | 申请用户 |
| `proposed_data` | JSON | NOT NULL | - | 提议修改的字段及新值，JSON格式 |
| `status` | VARCHAR(20) | NOT NULL | 'pending' | 状态：pending、approved、rejected |
| `admin_id` | INT | FK → users.id, NULL | - | 审核管理员 |
| `created_at` | DATETIME | AUTO_NOW_ADD | - | 创建时间 |
| `updated_at` | DATETIME | AUTO_NOW | - | 最后更新时间 |
| `is_active` | BOOLEAN | DEFAULT TRUE | TRUE | 是否启用（软删除） |

**索引：**
- `idx_edit_requests_status_created` (`status`, `created_at`) - 审核列表查询

| 字段名 | 类型 | 约束 | 默认值 | 描述 |

| `url` | VARCHAR(255) | NOT NULL | - | 图片访问路径（相对路径或 CDN 地址） |
| `entity_type` | VARCHAR(32) | NOT NULL | - | 关联实体类型：shop、menu_item、comment、question 等 |

| `extra` | JSON | NULL | - | 扩展字段，存储图片宽高、文件大小、alt 文本等元数据 |
| `created_at` | DATETIME | AUTO_NOW_ADD | - | 创建时间 |
| `updated_at` | DATETIME | AUTO_NOW | - | 最后更新时间 |
| `is_active` | BOOLEAN | DEFAULT TRUE | TRUE | 是否启用（软删除） |

**索引：**
- `idx_images_entity_type_id` (`entity_type`, `entity_id`) - 多态关联查询

**设计说明：** 采用多态关联（`entity_type` + `entity_id`），可灵活关联到不同实体。

## 设计问题与建议

### 1. 循环导入问题 ⚠️

**问题描述：**
- `users.py` 导入了 `Shops`（用于 `Favorites` 的外键）
- `shops.py` 导入了 `Users`（用于 `Ratings`、`Comments` 的外键）
- 这形成了循环导入依赖

**建议方案：**
使用字符串引用代替直接导入，Tortoise ORM 支持通过 `"models.ModelName"` 字符串形式定义外键，可避免循环导入。

```python
# 修改前
from .shops import Shops
shop = fields.ForeignKeyField("models.Shops", ...)

# 修改后（直接使用字符串，无需 import）
shop = fields.ForeignKeyField("models.Shops", ...)
```

---

### 2. 主键类型不一致 ⚠️

**问题描述：**
| 主键类型 | 使用的表 |
|----------|----------|
| `IntField` | users, activities, favorites, messages, shops, menu_items, ratings, dict_types, dict_data, shop_dict_rel, comments_likes, shop_edit_requests |
| `BigIntField` | comments, user_behavior_logs, images |

**建议方案：**
- 对于数据量可能快速增长的表（评论、日志、图片），使用 `BigIntField` 是合理的
- 建议在文档中明确说明：预计数据量超过 20 亿的表使用 `BIGINT`，其余使用 `INT`
- 当前设计基本合理，但建议统一规范：**核心业务表使用 INT，日志/流水类表使用 BIGINT**

---

### 3. 缺少字段约束 ⚠️

**问题描述：**

| 表 | 字段 | 问题 |
|----|------|------|
| `users` | `phone` | 标记为 `NOT NULL` 但无 `UNIQUE` 约束，可能导致重复手机号 |
| `users` | `email` | 标记为 `NOT NULL` 但无 `UNIQUE` 约束，可能导致重复邮箱 |
| `ratings` | `score` | 无 1-5 范围约束，代码注释提到但未实现 |
| `shop_edit_requests` | `status` | 无枚举约束，可能插入非法状态值 |

**建议方案：**

```python
# users.py - 添加唯一约束
phone = fields.CharField(max_length=20, unique=True, null=False, description="手机号")
email = fields.CharField(max_length=100, unique=True, null=False, description="电子邮箱")

# shops.py - 添加评分范围验证
score = fields.IntField(description="评分值（1-5）")
# 在 Meta 中或通过 validator 实现范围约束

# 或使用 IntEnumField
from tortoise.fields import IntEnumField

class RatingScore(IntEnum):
    ONE = 1
    TWO = 2
    THREE = 3
    FOUR = 4
    FIVE = 5

score = IntEnumField(RatingScore, description="评分值")
```

### 4. 多态关联设计 📝

**问题描述：**
以下表采用多态关联（`entity_type` + `entity_id` / `target_type` + `target_id`）：
- `images`
- `activities`
- `user_behavior_logs`

**优点：** 灵活，可关联任意实体
**缺点：** 无法在数据库层面建立外键约束，数据一致性依赖应用层保证

**建议：** 当前设计可接受，但建议在应用层增加验证逻辑，确保 `entity_type` 和 `entity_id` 的有效性。

### 5. 冗余字段维护 📝

**问题描述：**
`shops` 表包含多个冗余字段：
- `view_count` - 总浏览量
- `favorite_count` - 收藏数
- `comment_count` - 讨论数
- `average_rating` - 平均评分

**建议：**
- 冗余字段设计合理，可减少 JOIN 查询，提升读取性能
- 必须确保在相关操作时同步更新这些字段（使用信号/触发器/事务）
- 建议在 `favorites`、`comments`、`ratings` 的增删操作中添加更新逻辑

### 7. 软删除一致性 📝

**问题描述：**
所有表都继承了 `SoftDeleteMixin`，包含 `is_active` 字段。

**建议：**
- 对于日志类表（`user_behavior_logs`），软删除可能不必要，日志通常只追加不删除
- 对于关联表（`shop_dict_rel`、`comments_likes`、`favorites`），软删除可能导致唯一约束冲突（删除后重新创建会违反 `unique_together`）
- 建议：关联表可考虑不使用软删除，或在唯一约束中加入 `is_active` 字段

### 8. 命名规范 📝

**当前命名：**
- 表名：全部使用复数形式（`users`、`shops`、`comments`）✅
- 模型名：部分使用复数（`Users`、`Shops`、`Comments`），部分使用单数（`Menu`、`Images`）

**建议统一：**
- 模型名统一使用单数形式（符合 ORM 惯例）
- `Menu` → `MenuItem`（已在注释中说明原为 MenuItems）
- `Images` → `Image`
- `DictTypes` → `DictType`
- `DictData` → `DictDatum` 或保持 `DictData`（集合名词可接受）
