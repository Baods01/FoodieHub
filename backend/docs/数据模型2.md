## 三、多对多关系的处理

### 1. **店铺与点餐方式** —— 需改为中间表

当前 `shops` 表有 `dining_methods` 字段（INT），指向 `dining_methods` 表。这意味着一家店铺只能有一种点餐方式，但实际场景中一家店铺可能同时支持“堂食”“外卖”“自取”等多种方式。

**解决方案**：删除 `shops.dining_methods`，新建中间表：

```sql
CREATE TABLE shop_dining_methods (
    id INT PRIMARY KEY AUTO_INCREMENT,
    shop_id INT NOT NULL,
    dining_method_id INT NOT NULL,
    sort_order INT DEFAULT 0,  -- 用于前端展示顺序
    FOREIGN KEY (shop_id) REFERENCES shops(id) ON DELETE CASCADE,
    FOREIGN KEY (dining_method_id) REFERENCES dining_methods(id),
    UNIQUE KEY (shop_id, dining_method_id)
);
```

### 2. **店铺与类别** —— 根据业务决定是否多对多

你的 `shops` 表有 `category` 字段指向 `categories`。如果业务允许一个店铺属于多个类别（例如“中餐”+“快餐”），则同样需要中间表。

**如果支持多类别**：

删除 `shops.category`，新建：

```sql
CREATE TABLE shop_categories (
    id INT PRIMARY KEY AUTO_INCREMENT,
    shop_id INT NOT NULL,
    category_id INT NOT NULL,
    FOREIGN KEY (shop_id) REFERENCES shops(id) ON DELETE CASCADE,
    FOREIGN KEY (category_id) REFERENCES categories(id),
    UNIQUE KEY (shop_id, category_id)
);
```

**如果仅支持单类别**，则 `shops.category` 保留外键即可。

- **用户收藏店铺**：`favorites` 表充当中间表，已正确。

## 五、自关联表（评论）的外键处理

`comments` 表中的 `parent_id` 和 `root_id` 都指向 `comments.id`。建议添加外键约束：

```sql
ALTER TABLE comments
ADD FOREIGN KEY (parent_id) REFERENCES comments(id) ON DELETE CASCADE,
ADD FOREIGN KEY (root_id) REFERENCES comments(id) ON DELETE CASCADE;
```

但需注意：

- 当 `parent_id` 或 `root_id` 为 NULL 时，外键允许。
- 循环引用应通过应用层避免（例如不允许将已删除的评论作为父级）。
